# IronTide ‚Äî Codebase Cleanup Directions (design drift removal)

> **For:** Claude Code, working in `alan09086/irontide`.
> **Goal:** collapse the v0.1 multi-skin / multi-theme / multi-layout machinery down to the **one locked direction** (`DESIGN-SYSTEM.md` + `DESIGN.md`): emerald-dark, KDE, L1 3-pane, compact default. Remove the dead axes entirely ‚Äî they are drift, not features.
> **Visual truth:** the prototype in this handoff (`IronTide.html`, `styles/tokens.css`, `components/*`).

The design was authored across three skins √ó light/dark √ó three layouts. We shipped exactly one of those and the rest has rotted into a maintenance tax (a Python codegen step, a CI drift gate, 6 palette tables, 4 enums, dead config fields). This document is the surgical removal plan, file by file.

Do this as **one focused PR** with green tests at the end. Work top-down: tokens first, then the Rust state machine, then prefs UI, then scripts, then docs.

### 7a. Menu bar ‚Äî DO NOT use the native `MenuBar` widget (this is the thing that kept breaking)

The intended menu look is fully styled (emerald checks, mono accelerators, Breeze panels) ‚Äî see **`Menu Reference.html`** (all menus + submenus rendered open) and **`DESIGN.md ¬ß1a`** (item tables). Slint's built-in `MenuBar { Menu { MenuItem } }` hands rendering to the **platform's native menu system**, which ignores every color token ‚Äî you cannot make it look like the reference. That mismatch is almost certainly why prior attempts "weren't done right."

**Build a custom menu instead:**
1. A `MenuBarStrip` row of `MenuLabel` buttons (`File ‚Ä¶ Help`), Breeze flat hover (`--bg-hover`), 28px tall, `--bg-1`.
2. Clicking a label opens a `PopupWindow` (Slint's popup is freely styleable) containing a `VerticalLayout` of rows.
3. A reusable `MenuItemRow` component: `[16px mark gutter] [label, Inter 12.5] [spacer] [accelerator, mono 11 --fg-3] [chevron if submenu]`, 27px tall, hover `--bg-hover`, radius `--r-sm`. Variants: `check` (emerald ‚úì when on), `radio` (emerald dot when active), `submenu` (chevron + nested `PopupWindow` on hover), `separator` (1px `--divider`).
4. Drive it from a single Rust/Slint data model mirroring `MENU_MODEL` in `chrome.jsx`. Toggle/radio marks bind to live state (`Tokens` density/sidebar, tweak flags).
5. Register all accelerators in **`accel.rs`** independently, so `Ctrl+K`, `Ctrl+I`, `Ctrl+,` etc. fire whether or not a menu is open. The menu items only *display* the accelerator; `accel.rs` owns the binding.

Acceptance: the running app's menus are pixel-comparable to `Menu Reference.html` ‚Äî emerald checks/dots, mono shortcuts, hover wash, working submenus ‚Äî not stock OS menus.

### 7b. Text overflow ‚Äî never let text spill into adjacent rows/fields

This is a recurring defect (in both the prototype and the Slint port): a **fixed-height row or fixed-size cell holds text that wraps**, and the second line draws on top of the row/field below. Apply these rules everywhere:

**Slint `Text`:**
- In any fixed-height row, table cell, button, menu item, status bar, or label: set `wrap: no-wrap;` and `overflow: elide;` so long content ellipsizes instead of wrapping. (`elide` renders the `‚Ä¶`.)
- Multi-line text (comments, hints, descriptions) is the *only* place `wrap: word-wrap` is allowed ‚Äî and its container must size to content (no fixed height).
- Put a `clip: true` on fixed-height row containers so nothing can paint outside them.

**Layout:**
- Don't pin a `height` on a container whose text can grow; either elide (single-line) or let the layout compute height (`VerticalLayout` with `min-height`, not fixed `height`).
- Right-aligned numeric columns (speeds, sizes, ratios): `horizontal-alignment: right` + `overflow: elide` so a too-wide value clips on the left edge of its own cell, never overlapping the neighbour.
- Reserve a fixed gutter for accelerators/badges so the label has a hard right boundary to elide against.

The prototype now does this: torrent-table rows are `overflow: hidden` + every cell is single-line; `Btn`/`Chip` are `white-space: nowrap`; the `Toggle`/checkbox control puts the switch in a `flex-shrink: 0` box and lets the label wrap so the **row grows** instead of the second line painting over the next control; and **every multi-column table row** (torrent list, Search results, RSS items) pins its fixed columns with `flex-shrink: 0` and lets only the name/title column flex + elide ‚Äî otherwise the fixed columns shrink and their values collide (this was the Search "PEERS"/"ENGINE" ‚Üí "312LinuxTracker" overlap). Mirror it in `torrent_row.slint`, `sidebar_item.slint`, the prefs field rows, every dialog, the tool-page tables, and **every checkbox/toggle label**. Acceptance: resize the window narrow ‚Äî text must clip/elide or grow its row, never overlap.
